# Switch the webpage to render watermarked property images

A self-contained brief for the Claude instance running in `~/Desktop/vestawebpage`.

I need to switch this webpage from rendering clean property images to rendering watermarked versions. The watermarked files **already exist in S3** — they're generated by `vesta-crm` at portal-publish time and at website-publish time, with a deterministic key pattern that lets you construct URLs with no API call and no database join.

## Key facts

1. **Watermarked file location** — deterministic, no DB or API call required:

   ```
   https://vesta-prod.s3.{AWS_REGION}.amazonaws.com/
     accounts/{accountId}/watermarked/{propertyId}/image_{imageOrder}.jpg
   ```

2. **There is ONE watermarked file per source image.** No thumb/med/full variants of the watermarked version. Just one ~100–150 KB JPEG (vesta-crm just optimized this — it's now sourced from the `medUrl` variant at q82, ~1200 px wide). Don't look for `image_3_med.jpg` etc. — they don't exist on the watermarked side.

3. **The `(accountId, propertyId, imageOrder)` tuple is 1:1 with clean `propertyImages` rows.** Same `imageOrder` = same image, just branded.

4. **Watermarked files may NOT exist for every listing.** They are only generated for listings where watermarking is enabled (per-account flag `watermarkEnabled`, optionally overridden by the per-listing `watermarkProps.enabled` field on the `listings` table). You need a fallback strategy when the file is absent.

## What to change

Replace the rendered URL of property images with the watermarked URL, **except**:

- **Card thumbnails (list/grid views): KEEP using the clean `thumbUrl` variant.** Watermarks at 400 px are illegible and the bytes aren't worth it. Cards live on the agency's own domain so brand attribution is implicit. Don't waste a network round trip on a watermarked thumbnail nobody will be able to read.

- **Gallery default view (med-tier surfaces): switch to the watermarked URL.**

- **Fullscreen viewer / lightbox (full-tier surfaces): switch to the watermarked URL.**

- **AI before/after slider: switch BOTH sides (parent and derived) to their respective watermarked URLs.** See "Constraint 2" below.

## Constraints / things you must not break

### 1. Image tags (`cocina`, `salon`, `plano`, `video`, `youtube`, virtual-tour)

The webpage groups, filters, or specially renders images by tag. The tag lives on the `propertyImages` row in the `imageTag` column and is **unchanged**. Just keep reading it from the row — only the rendered URL field changes. **Don't break the tag-based grouping logic.**

### 2. AI before/after slider (parent image relationship)

When a `propertyImages` row has `originImageId` set, it's an AI-derived image (enhanced/renovated/etc.) and the webpage shows a before/after slider comparing it to its parent. **Both sides of the slider need to use watermarked URLs**:

- The derived image's watermarked URL uses **its own** `imageOrder`.
- The parent image's watermarked URL uses the **parent's** `imageOrder` (look up the parent row by `propertyImageId = originImageId`).
- Both files live under the **same** `accountId/propertyId` path (AI derivatives stay on the same property).

The vesta-crm watermark pipeline processes ALL active rows in `propertyImages`, including AI-derived ones, so both watermarked files exist at their respective `image_{imageOrder}.jpg` keys. You don't need special-casing — just construct each URL from the row you're rendering.

### 3. 1:1 ordering

Clean and watermarked files share the same `imageOrder`. **Don't reorder, don't remap, don't try to be clever.** The watermarked URL is always derivable from the clean row's `imageOrder`.

### 4. Fallback when watermarked file is absent

For listings where watermarking is disabled (per-account or per-listing), the watermarked file simply does not exist in S3. You need either:

- **(a)** Read the listing's effective watermark config server-side and decide per-listing whether to use the watermarked URL or the clean variant. Free if the listing data is already loaded.

- **(b)** Fall back on 404/missing image to the clean variant at render time. Costs an extra request when watermarks are off.

**Pick (a) if you can.** Resolve which is feasible during planning.

## Notes on the brief

**On the slider constraint:** Double-checked while drafting. The vesta-crm watermark pipeline (`getPropertyImages` → `processWatermarkedImagesWithCache`) iterates over all active rows in `propertyImages` for a given property, with no filter on `originImageId`. So AI-derived rows do get their own `image_{imageOrder}.jpg` watermarked file at the same `accountId/propertyId` path. This means the slider "just works" — both the parent and the derived image have predictable watermarked URLs. The only thing you have to do is look up the parent row via `originImageId` to get its `imageOrder`, then construct the URL the same way as for any other image.

**On the fallback question:** Phrased as an open choice (read config vs catch 404) because I don't know how the webpage currently fetches data. If it loads listings server-side via Supabase or a vesta-crm endpoint, option (a) is essentially free — it just adds the `watermarkEnabled` field to the existing query. If the webpage is fully static-generated with no per-render server logic, option (b) is the only choice but adds an extra request on the failure path. You're in the right position to pick the right one.

**On the open questions:** Q3 (CDN in front of S3) is the one I'm most unsure of and most curious about — if there is a CDN, the cache invalidation story for "watermark config changed → re-render webpage" gets non-trivial, because vesta-crm regenerates the S3 file but the CDN still serves the old one until its TTL expires. Worth knowing before shipping. Surface it during planning rather than during deployment.

**On scope discipline:** The brief explicitly says "don't implement until I've reviewed the plan" and "ship in commits grouped by display surface." Both are anti-runaway-Claude guardrails — the change touches a lot of files and you'll want to review it surface-by-surface, not as one giant blob.

**One thing deliberately NOT included:** any code snippet for constructing the URL. The receiving codebase probably has its own conventions for building S3 URLs (env-var-driven region, central helper, maybe a CDN base URL prefix). Better to wire up the URL in whatever pattern the rest of the repo uses.
